## 1. 文档目标

本文档定义在线 SSH 平台第一版的系统总体架构、模块边界、核心数据流、关键运行链路与落地约束。目标是让后续开发、AI 辅助编码、数据库设计、接口设计与测试设计都有统一基线。

本文聚焦第一版 MVP：**个人版优先、密码/私钥登录、平台中转传输、操作审计优先、RBAC 预留**。

---

## 2. 系统目标

第一版系统需要满足以下核心能力：

1. 用户登录平台并保存自己的主机配置与凭据
2. 用户通过浏览器建立 SSH 终端会话
3. 用户浏览远程文件系统并进行基础文件管理操作
4. 用户通过浏览器拖拽上传文件到远程服务器
5. 用户从远程服务器下载文件到本地
6. 用户查看传输任务状态，并支持暂停、恢复、取消、重试
7. 用户查看自己的登录、连接、文件操作与传输记录

---

## 3. 总体架构

### 3.1 逻辑分层

系统采用前后端分离架构，按职责划分为以下层次：

1. **Web UI 层**
   - 浏览器页面
   - 终端渲染
   - 文件管理器 UI
   - 任务与日志展示
   - 工作区 / 标签页状态

2. **应用 API 层**
   - 用户认证
   - 主机管理
   - 文件管理 API
   - 任务管理 API
   - 审计查询 API

3. **实时通道层**
   - 终端 WebSocket
   - 任务进度推送 WebSocket / SSE

4. **执行层**
   - SSH 会话执行
   - SFTP 文件浏览与操作
   - 文件上传 / 下载执行器
   - 传输任务调度器

5. **持久化层**
   - PostgreSQL：业务主数据、任务、审计、布局
   - Redis：会话缓存、短状态、去重锁、任务协调
   - 本地对象存储 / 临时目录：上传分片与中间文件

## 4. 模块边界

### 4.1 前端模块

#### auth-app

负责登录、登出、会话恢复、用户信息获取。

#### host-manager

负责主机列表、主机编辑、测试连接、认证方式配置。

#### terminal-workspace

负责标签页、终端实例、工作区布局、终端连接状态。

#### file-browser

负责远程目录浏览、路径导航、列表视图、右键菜单、搜索与文件操作入口。

#### transfer-center

负责传输任务列表、状态、速度、暂停/恢复/取消、失败展示。

#### audit-viewer

负责日志查询、筛选、详情展示。

#### ui-shell

负责导航、主题、国际化、布局持久化。

### 4.2 后端模块

#### auth-service

- 用户登录鉴权
- 会话签发与刷新
- 登录日志记录

#### credential-service

- SSH 密码 / 私钥加密存储
- 建立连接前凭据解密
- 密钥版本管理

#### host-service

- 主机 CRUD
- 指纹记录
- 最近连接时间更新

#### terminal-gateway

- WebSocket 终端通道
- SSH Session 与 PTY 绑定
- 输入输出转发
- resize / keepalive / 断线处理

#### file-service

- 目录列举
- 文件属性查询
- mkdir / rename / delete / chmod
- 文件下载准备

#### transfer-service

- 上传初始化
- 分片写入
- 平台中转上传/下载任务
- 状态推进
- 偏移恢复

#### transfer-worker

- 实际执行传输任务
- 读取源、写入目标
- 更新进度与审计

#### audit-service

- 记录结构化事件
- 查询与筛选
- 后续扩展命令审计

## 5. 核心运行链路

### 5.1 用户登录链路

1. 用户在 Web 登录页输入账号密码
2. 前端通过 HTTPS 调用登录 API
3. 后端校验账号密码
4. 后端生成会话 token / cookie
5. 记录登录事件与设备信息
6. 前端进入主控制台

### 5.2 SSH 终端连接链路

1. 用户点击某台主机“连接终端”
2. 前端调用 terminal bootstrap，获取 `session + websocket`
3. 前端携带登录 session cookie 和短生命周期 attach token 建立 WebSocket 终端通道；后端在 token 校验通过后兼容 Host 重写反代
4. 后端读取主机配置与凭据
5. 后端校验 host fingerprint
6. 后端创建 SSH Client + Session + PTY
7. SSH 输入输出绑定到 WebSocket
8. 记录 host_connect 事件
9. 用户进行交互操作
10. 断开时记录 host_disconnect 事件

只读分享观看链路使用独立 `/ws/terminal/share`，前端先通过公开分享 token 换取短期 viewer token，再建立只读 WebSocket；viewer token 校验通过后同样兼容 Host 重写反代。

### 5.3 远程文件浏览链路

1. 用户在文件页选择某台主机
2. 前端请求 `/files/list`
3. 后端建立或复用 SFTP 连接
4. 读取目录项与元数据
5. 返回给前端渲染
6. 记录 `file_list` 事件（可按策略降低频度）

### 5.4 本地拖拽上传链路

1. 用户从本地拖拽文件到浏览器
2. 前端调用上传初始化接口，创建任务
3. 前端按 chunk 上传到平台暂存区
4. 平台记录每个 chunk 状态
5. 平台分片合并或按偏移写入中间文件
6. transfer-worker 通过 SFTP 发送到远程 `.part` 文件
7. 完成后 rename 为目标文件
8. 更新任务状态，记录审计日志

### 5.5 远程文件下载链路

1. 用户选择远程文件并点击下载
2. 后端创建下载任务
3. transfer-worker 从远程读取文件
4. 可选：先落到平台临时文件，再提供下载 URL
5. 前端触发浏览器下载
6. 用户通过浏览器保存到本地
7. 记录下载开始、成功、失败日志

## 6. 第一版部署形态

### 6.1 推荐部署

第一版建议采用单体服务优先：

- 一个前端应用
- 一个后端主服务
- 一个 PostgreSQL
- 一个 Redis
- 一个本地临时文件目录或对象存储桶

### 6.2 原因

- MVP 阶段重点是把 SSH、文件传输、安全与审计打通
- 复杂微服务拆分会提高运维与调试成本
- 单体但模块边界清晰，后续仍可拆分 transfer-worker / audit-service

---

## 7. 非功能要求

### 7.1 性能

- 单用户可同时保持多个终端会话
- 支持多个并发传输任务
- 大文件传输需稳定，允许中断后恢复
- 大目录列表需使用分页 / 懒加载 / 虚拟列表策略

### 7.2 稳定性

- 终端通道断开要明确提示
- 传输任务状态不可丢失
- 服务重启后可识别未完成任务并恢复/标记异常

### 7.3 安全

- 敏感凭据加密存储
- 全链路 TLS + SSH
- 主机指纹校验
- 操作日志可追踪

### 7.4 可扩展性

- 预留 RBAC 字段与资源归属字段
- 预留命令审计能力
- 预留远端到远端直传能力

## 9. AI 辅助开发建议

为了便于 AI 生成代码，后续文档与代码仓库建议遵循以下原则：

1. 每个模块必须有明确输入输出与职责
2. DTO、数据库模型、领域模型分层定义
3. 状态机必须文档化
4. 所有审计事件名固定枚举化
5. 所有错误码固定枚举化
6. 关键链路有时序图或步骤说明
7. 每个 API 都有请求、响应与失败示例

## 10. 本文结论

第一版最优路线是：**单体架构起步，围绕终端、文件传输、安全、审计建立清晰模块边界；先保证稳定与安全，再逐步增强布局、命令审计与协作能力。**
